System Breakdown

AI authors structure. The system owns execution.

This is not just “AI generates content.” It is a Codex-assisted development loop and a compiler-like pipeline for turning generated structure into runtime systems.

Codex helps build the tooling, schemas, validation rules, editor scripts, and workflow doctrine. Structured AI output then flows through those tools into validated engine artifacts and stateful runtime behavior.

The result is AI-assisted system authoring without giving AI runtime authority.

System Breakdown

How This System Works

This system uses AI at two levels: Codex helps build the tooling and rules that operate the pipeline, while prompt-driven generation produces structured candidate content. That generated structure is validated, compiled into engine-side artifacts, and registered into a runtime layer that owns behavior, state, and authority.

1. Architectural Framing

This is a compiler-like content-to-runtime pipeline wrapped around a simulation architecture. It is not a simple “AI generates game content” flow. It is a controlled authoring system where generated structure is compiled into engine-owned representations and executed by a stateful runtime.

There are two connected systems:

  • an authoring pipeline that transforms generated structure into validated engine artifacts
  • a runtime execution layer that owns behavior, state, and authority

That split is the defining architectural move.

2. AI-Assisted Development Loop

The system is AI-assisted at two levels.

  • First, Codex is used throughout development to help generate and revise editor tooling, schemas, validation rules, runtime bridges, workflow scripts, and doctrine. It accelerates the construction of the machinery that processes structured content.
  • Second, prompt-driven generation produces candidate scenes, identities, actions, conditions, and authored relationships. Those outputs are not executed directly. They pass through the pipeline built around schemas, validation, tooling, and runtime ownership.

This creates a compounding loop: AI helps build the tools, then structured AI output flows through those tools into runtime behavior.

3. Core Design Principles

  • The first principle is separation of generation from authority. AI can propose structure, but it never owns execution. Execution belongs to the runtime.
  • The second is structure before behavior. Nothing enters runtime as prose or loose intent. It has to become explicit data first.
  • The third is validation as a control boundary. Validation is not decoration. It is the gate that prevents malformed generated output from becoming executable state.
  • The fourth is observability as part of the system contract. Generated systems are only useful if they can be inspected, debugged, and verified after generation.
  • The fifth is authoring/runtime asymmetry. The authoring side can be flexible and AI-assisted. The runtime side remains explicit, constrained, and deterministic.

4. System Architecture

Prompt layer

The entry point is prompt-driven generation. The model produces authored content shaped toward defined schemas instead of prose. The important output is not description, but structured declarations the rest of the pipeline can evaluate.

Structured data layer

Generated output is normalized into structured data with explicit fields, allowed shapes, and validation boundaries. This is where ambiguity is removed. Identities, conditions, actions, and dependencies become machine-checkable rather than interpretive.

Build/tooling layer

Validated data moves into Unity-side import and generation tooling. This layer handles import, schema checks, asset generation, registration, and build-time validation. Scenes, prefabs, and supporting runtime definitions are produced from structured input rather than manually assembled one-offs.

Runtime layer

At runtime, generated content resolves into stateful systems. Objects, actors, and simulation components operate over runtime-owned data. The prompt is no longer part of execution. The runtime only knows validated state, registered systems, and executable behavior.

Prompt -> structured data -> validation/import -> Unity generation -> runtime registration -> playable behavior

5. Runtime Behavior

The runtime is stateful and system-driven. Once content is generated and registered, behavior is owned by the runtime layer, not by the original prompt and not by loose scene metadata.

State ownership is explicit. Stats, conditions, resources, and behavior-driving values exist as runtime data and are updated through controlled pathways. Systems interact over time through commands, ticks, or both.

For example, a generated health system does not exist as text or metadata. It resolves into runtime-owned state such as Health and MaxHealth, updated through validated commands and system ticks. Effects like damage or healing are applied through controlled pathways, not direct mutation.

That is what makes the output playable. The runtime is executing system logic over owned state, not replaying authored text.

6. Boundaries and Control Points

Authority lives in four places:

  • schemas define the allowed shape of authored output
  • validation enforces admissibility before build/runtime
  • tooling converts structured inputs into engine-side artifacts
  • runtime systems own actual execution and state mutation

The critical boundary is between structured data and executable runtime content. That is where generated output stops being suggestion and starts being admitted into the system.

Another boundary exists between authored structure and live state. Authored inputs define what can exist. Runtime defines what is currently true.

AI can influence the system upstream, but it cannot bypass downstream authority.

7. Tooling, Control, and Observability

The goal is not just to generate systems, but to operate them.

The pipeline includes import validation, runtime inspection, and debugging surfaces that expose how generated systems behave once they are live. Live state editing, runtime inspection, and validation allow systems to be exercised under real conditions. Stats, conditions, and cross-system data can be modified and observed without regeneration.

Observability is part of the architecture. Logs, validation artifacts, and inspectable runtime state make behavior explainable after the fact, not just visible in the moment.

8. Failure Modes This Avoids

This architecture is designed to avoid:

  • direct execution of underspecified generated output
  • hidden state mutation through content-layer artifacts
  • runtime behavior that cannot be traced back to validated inputs
  • brittle one-off glue code between prompts and engine behavior
  • pipelines where debugging requires re-reading prompt text instead of inspecting runtime state
  • toolchains where AI output contaminates authority boundaries

In short, it avoids treating generated text as if it were already a system.

9. Why This Works

This works because it separates generation from authority.

Codex can accelerate the creation of tooling, rules, scripts, and workflows. Prompt-driven generation can accelerate content and system authoring. But the system is responsible for validation, generation, registration, and runtime behavior.

This shifts AI from execution to authoring.

The model proposes structure. The system enforces rules.

10. What This Enables

This architecture enables fast prototyping without sacrificing runtime control. New scenes, systems, and behaviors can be generated quickly, but they still move through the same validation, build, and registration path each time.

It also enables repeatability. Because the pipeline is structured, it can scale across prompts, content sets, and prototypes without collapsing into manual cleanup and one-off fixes.

The result is not generated content. It is controllable, stateful software with defined behavior, ownership, validation, and runtime authority.

Runtime Architecture

Command-Driven Simulation Engine

A gameplay architecture where Unity is not the source of truth and all runtime state is owned by a central simulation. Scene objects submit commands into an authoritative plain C# simulation that owns all runtime state.

  • Central GameState owns runtime truth
  • Unity objects act as input, scene, and presentation bridges
  • Commands are validated before mutating state
  • Ordered tick systems advance simulation over time
  • Logging and validation capture commands, ticks, and state changes

This is a simulation engine, not a Unity scene.

World Simulation

Persistent World Systems

A shared-world simulation architecture where agents, schedules, identity, law, security, economy, body state, buffs, combat AI, and observability operate as coordinated subsystems.

  • Identity-driven agents connect schedule, access, law, economy, and inspection
  • World time drives behavior instead of one-off scripted sequences
  • Separate AI systems handle civilian scheduling and combat behavior
  • Player actions can propagate into crime, escalation, and security response

This is a simulated world, not a scripted experience.
