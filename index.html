AI-Assisted Engineering

I make the repository control Codex, not the other way around

Most people use Codex like a chat window: ask for code, accept a patch, hope it works. I built a repo workflow where Codex has to read project rules, take one written task, change the correct files, run checks, and record what happened before moving on. That is the difference between using AI as a code generator and using AI inside an actual engineering process.

The problem I solved

AI coding gets messy when the prompt is the only thing controlling the work. The model may edit the wrong file, forget earlier decisions, create a duplicate system, or produce code that looks right but breaks the project later.

My solution is to make the repository control the AI. A prompt starts the request, but the repo tells Codex how to work: what rules to read, where the responsible code lives, what files are off limits, what checks count as done, and what notes must be updated for the next session.

Project rules
I call these doctrine in my repos. Plain meaning: written rules Codex must follow before changing code.
Startup file
AGENTS.md is the first file Codex reads. It points Codex to the project rules, task list, architecture notes, and decisions.
Task list
A prompt becomes a written task. The task says what problem is being solved, what part of the project to inspect, and what must be shown before the task is complete.
Architecture notes
These are files inside the repo that explain what systems exist, what they do, and how they connect.

The system I built around Codex

I split the workflow into reusable engineering infrastructure and project-specific work. The reusable part defines how Codex starts, how tasks are tracked, how generated files are protected, how checks run, how failures are logged, and how old or unused code gets cleaned up. The project-specific part contains the actual product code, design notes, generated asset sources, runtime logs, and current task state.

That structure matters because Codex no longer has to infer the whole codebase from a chat. It enters through the same path every time, reads the current rules, uses the current architecture notes, works on one task, and leaves the repo in a state the next Codex run can understand.

Where most AI coding workflows go wrong

A lot of teams are still treating Codex like a faster autocomplete loop. That can work for small fixes, but it breaks down when the codebase has generated files, long-running decisions, runtime behavior, editor tooling, and systems that need to stay consistent over multiple sessions.

Common pattern Why it breaks My workflow
Prompt to code to quick fix to repeat. The same problems come back because nothing records what was learned or what rule should change. The prompt becomes a written task. Codex changes the right files, runs checks, and records the result for the next session.
Cram every Codex instruction into AGENTS.md. The file turns into a giant manual. It gets hard to scan, easy to contradict, and painful to keep current. AGENTS.md stays a startup map. It tells Codex what to read first and where the real project rules, tasks, decisions, and architecture notes live.
Let chat history be the project memory. The next session loses important context or has to rediscover it from scratch. Important context lives in repo files: rules, decisions, architecture notes, task history, logs, and check results.
Ask for a whole feature in one loose prompt. Codex may touch too much code, create duplicate systems, or solve the visible symptom instead of the real problem. The work is broken into clear tasks. Codex checks the existing system first, then changes the part of the code responsible for the behavior.
Patch generated Unity files by hand. The next generation pass can overwrite the fix, or the generated output stops matching the source data. Codex changes the generator, factory, authored data, or sync tool that creates the output.
Treat a clean-looking patch as done. Code can look reasonable while runtime behavior is still broken. The check has to match the risk: build checks for code, parser checks for data, runtime logs for behavior, and screenshots or manual verification for visual work.

How a request becomes working code

  1. Write the request down as a task. Codex does not just start editing. The request is captured as a task with a goal, relevant files or systems, and a definition of done.
  2. Read the project instructions. Codex loads the startup file, project rules, architecture notes, current task list, and recent decisions before it touches code.
  3. Find the code that should handle the problem. Before adding something new, Codex checks whether an existing system already owns that behavior.
  4. Change the files that create the behavior. If a Unity scene, prefab, material, or data file is generated, Codex changes the source that creates it instead of hand-editing the generated copy.
  5. Run the checks that match the risk. Code changes need build checks. Data changes need parser checks. Runtime bugs need logs. Web changes need link and render checks.
  6. Record what changed. Codex updates the task history and architecture notes so the next session knows what happened and why.

What the workflow prevents

Wrong-file fixes Codex is directed to the source files that create the behavior, not generated output that will be overwritten later.
Duplicate systems Codex has to check existing systems and extension points before inventing another way to solve the same problem.
Lost context Important decisions live in repo files Codex can read, not only in one chat transcript.
Unverified changes Work is not treated as done until the relevant check, log, screenshot, or human verification is recorded.

Examples from real development work

These examples show the difference between asking an AI for code and making an AI operate inside an engineering workflow. Codex had project rules, runtime logs, existing systems to respect, and checks it had to complete.

Fixing first-person object throwing

Context
A first-person controller let the player pick up small physical objects, hold them in front of the camera, drop them, or charge and throw them.
Problem
Thrown objects looked like they disappeared or broke immediately instead of flying forward.
How Codex diagnosed it
Runtime logs showed the object was being released from the player root near the floor, not from the held position in front of the camera.
What Codex changed
Codex fixed the existing interaction and pickup-object code so the object releases from the carried pose, wakes up as a real physics body, restores gravity and collisions, and briefly ignores the player so it does not collide instantly.
How the work was checked
Codex ran build checks, verified project data files still parsed, scanned for unused code, checked input-system rules, and updated the architecture notes.
Why it matters
The fix solved the actual physics problem without creating a second inventory, throwing, or item system.

Stabilizing procedural creature movement

Context
The project used small modular creatures with procedural legs. Their feet were placed at runtime instead of using fixed animation clips.
Problem
Creatures were getting stuck, taking tiny repeated steps, or dragging legs toward unreachable targets.
How Codex diagnosed it
Runtime traces showed repeated leg-support failures and bad foothold fallback events. That narrowed the issue to gait scheduling and foot target selection.
What Codex changed
Codex updated the procedural leg controller and animator so legs preserve support, avoid unreachable targets, and keep foot placement inside each leg's real movement range.
How the work was checked
Codex ran build checks, unused-code scans, input-system checks, data-file checks, and updated the architecture notes that describe the movement system.
Why it matters
The workflow used logs to target the responsible movement code instead of rewriting the whole creature system from a guess.

Protecting generated Unity assets

Context
Some Unity objects are produced from source data and generator code. That is useful, but it means generated output should not be manually patched as if it were the real source.
Problem
Reusable equipment art was being created only as scene output, which made future generation runs risky because edits could be overwritten.
How Codex diagnosed it
The rules showed the boundary was wrong: reusable art needed to be produced as project assets before scene placement, not treated as one-off scene objects.
What Codex changed
Codex moved responsibility into the generator and sync source so meshes, materials, textures, and prefabs are produced from authored factory code before scenes use them.
How the work was checked
The generated-file rule was documented, architecture notes were updated, and blocked editor work was recorded instead of being hidden or marked done.
Why it matters
The workflow protected the production pipeline instead of making fragile hand edits that would disappear later.
